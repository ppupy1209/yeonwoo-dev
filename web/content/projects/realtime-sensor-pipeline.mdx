## 개요
- 선박 내 온습도, 화재, 도어 센서 데이터를 외부 서버로 실시간 전송하는 시스템
- Worker Server에서 센서 데이터 수신 후 MQTT 메시지로 변환

## 문제

- API Polling 방식으로 센서 상태를 반복 조회하며 불필요한 HTTP 요청 누적
- 고정 스레드풀 100개와 무제한 큐 설정으로 트래픽 버스트 시 작업 적체 발생
- Context Switching 비용과 Queueing Delay 증가로 CPU 사용률 90%까지 상승

## 해결 전략

- API Polling 대신 MQTT 기반 발행 구조 적용
- 센서 데이터 수신 로직과 MQTT 전송 작업 분리
- CorePoolSize 100에서 8로 축소하고 Queue Capacity 1로 제한
- Queue 포화 시 오래된 전송 작업을 제거하는 정책으로 최신 상태 우선 처리
- MQTT 전환 효과와 스레드풀 튜닝 효과를 분리해 측정

## 기술 선택 이유

- MQTT
  - 작고 반복적인 센서 상태 메시지를 낮은 오버헤드로 지속 전송하는 요구에 적합
  - Pub/Sub 구조로 Worker Server와 외부 서버 간 결합도 감소
  - 온습도 데이터는 최신 상태 중심이라 QoS 0 적용 가능
  - 화재·도어 이벤트는 유실 허용도가 낮아 QoS 1 기준으로 재전송 보장
  - QoS 1 중복 수신은 센서 ID와 이벤트 시각 기준 멱등 처리로 흡수
  - QoS 2는 4-way handshake 비용 대비 외부 서버의 상태 기반 처리로 대체 가능

- Kafka 제외
  - 이벤트 영속성, 재처리, 대규모 스트림 처리보다 단순 실시간 전달 요구가 우선
  - Broker 운영, 파티션, 컨슈머 그룹 관리까지 포함하면 요구사항 대비 운영 복잡도 과도

## 튜닝 기준

- CorePoolSize 8
  - Worker Server 8코어 환경 기준으로 CPU 코어 수와 MQTT 전송 작업 특성을 함께 고려
  - 기존 100개 스레드 대비 Context Switching 비용 감소
  - 스레드 덤프에서 대기 스레드 비중이 높고, 실행 중 작업보다 전송 대기열 증가가 병목으로 확인

- Queue Capacity 1
  - 큐에 오래 쌓인 센서 데이터는 외부 서버 관점에서 이미 낡은 상태
  - 대기열 확대보다 최신 데이터 전달과 Queueing Delay 억제 우선
  - DiscardOldest 계열 거절 정책으로 대기 중인 이전 상태를 제거하고 신규 상태를 큐에 반영

## 검증

### 테스트 설정
- Worker Server 8코어 환경
- Locust로 센서 100개 동시 접속 재현
- 부하 산정 기준: 선박 1대 기준 센서 80개와 예비 확장 20개를 더한 100개 모델
- 센서별 초당 1회 전송, 전체 피크 초당 100건
- 2분 ramp-up 후 10분간 지속 부하 수행
- 측정 1: Polling 유지, 스레드풀만 100에서 8로 변경
- 측정 2: 스레드풀 8 유지, Polling에서 MQTT 발행 구조로 변경

### 측정 결과
- 스레드풀 튜닝 구간에서 피크 CPU 사용률 90%에서 60%로 감소
- MQTT 전환 구간에서 HTTP Polling 요청 제거와 외부 서버 결합도 감소 확인
- 큐 적체로 인한 MQTT 전송 지연 감소

## 배운 점
- 성능 개선 수치는 변경 변수를 분리해 설명할 때 설득력 확보
- MQTT QoS는 센서 중요도와 유실 허용 범위 기준으로 차등 적용
- QoS 1 선택 시 중복 수신을 정상 케이스로 보고 수신 측 멱등성까지 함께 설계
