## 개요
- 좋아요, 댓글, 조회 이벤트를 기반으로 당일 인기글 점수 집계
- Kafka 이벤트 스트림과 Redis Sorted Set을 이용한 Top 10 갱신

## 문제

- 매일 0시 전수 계산 방식으로 실시간 인기글 반영 불가
- 집계 시점에 타 서비스 API 호출이 몰리는 구조
- API 직접 호출 방식의 서비스 결합도와 이벤트 유실 위험
- at-least-once 발행 구조에서 중복 이벤트 소비 시 점수 중복 반영 가능

## 해결 전략

- 게시글 생성·삭제, 좋아요, 댓글, 조회 이벤트를 Kafka로 발행
- 인기글 Consumer에서 점수 계산 후 Redis Sorted Set 갱신
- Transactional Outbox로 DB 변경과 이벤트 발행 사이의 유실 구간 보완
- Redis dedup 키에 eventId 처리 이력을 저장해 중복 소비 방지
- Lua script로 dedup 기록과 ZSet 점수 갱신을 원자적으로 처리
- 실시간 기준을 Outbox relay 주기 포함 10초 내 반영으로 정의

## 기술 선택 이유

- Kafka
  - 여러 서비스에서 발생한 이벤트를 순차 로그로 수집하기에 적합
  - Consumer 장애 후 Offset 기반 재처리 가능

- Transactional Outbox
  - DB 커밋과 메시지 발행을 직접 한 트랜잭션으로 묶기 어려운 구조 보완
  - 미발행 이벤트 재전송으로 at-least-once 발행 보장
  - at-least-once 특성상 Consumer 멱등성 설계와 함께 사용 필요

- Redis Sorted Set
  - 인기글 점수 갱신과 Top N 조회를 같은 자료구조에서 처리 가능
  - 이벤트 ID dedup 이후 `ZINCRBY`를 수행해 중복 점수 반영 차단
  - dedup 키 TTL 24시간으로 당일 인기글 집계 범위와 메모리 사용량 균형 확보
  - 이벤트 100만 건 기준 dedup 키 메모리 약 100~150MB 수준으로 산정

## 검증

### 테스트 설정
- 게시글, 댓글, 좋아요, 조회 이벤트 토픽 구독
- Consumer 처리 완료 후 수동 커밋
- 미발행 Outbox 이벤트 10초 주기 재전송
- 부하 모델: 조회 이벤트 80 TPS, 좋아요·댓글 이벤트 20 TPS, 합산 100 TPS
- 동일 eventId 중복 발행 시 점수 1회 반영 여부 확인
- 동일 eventId 동시 소비 상황에서 Lua script 원자성 확인
- Outbox 저장 시점부터 Redis ZSet 반영까지 지연 시간 측정

### 측정 결과
- 0시 배치 집계에서 10초 내 스트림 반영 구조로 전환
- 집계용 API 직접 호출 제거
- 동일 eventId 중복 소비 시 인기글 점수 중복 증가 차단
- 100 TPS 기준 10분 부하에서 dedup TTL 키 증가량과 Redis 메모리 사용량 확인

## 핵심 흐름
- 비즈니스 트랜잭션 안에서 Outbox 이벤트 저장
- 커밋 이후 Message Relay가 Kafka로 발행
- Consumer가 Lua script로 dedup 키 저장과 Redis Top 10 ZSet 갱신을 한 번에 처리

## 배운 점
- at-least-once 발행은 Consumer 멱등성 없이는 정합성 보장 불가
- 실시간 처리는 표현보다 허용 지연 시간을 먼저 정의해야 함
- dedup 저장소는 TTL, 원자성, 메모리 비용까지 같이 설계해야 함
