## 개요
- Redis 조회 캐시 장애가 전체 조회 지연으로 확산되는 상황 차단
- Circuit Breaker와 Fallback으로 장애 경로를 원본 RDB 조회로 우회

## 문제

- Redis 장애 시 매 요청이 connection timeout과 retry 대기 시간만큼 지연
- 캐시 장애가 조회 서비스 전체 Latency 증가로 확산
- 장애 중에도 Redis 호출을 계속 시도하는 구조

## 해결 전략

- Redis 조회 호출을 Circuit Breaker로 감싸 장애율 감지
- Circuit Open 상태에서는 Redis 호출 생략 후 RDB로 폴백
- Cache Miss와 Circuit Open 예외를 분리해 fallback 경로 구분
- Fallback 트래픽이 RDB로 한 번에 몰리지 않도록 동시 요청 상한과 timeout 분리
- Circuit Open 중 Redis 재시도 폭주를 막고 Half-Open에서 제한된 요청만 회복 확인

## 기술 선택 이유

- Circuit Breaker
  - 장애가 반복되는 외부 의존성 호출을 빠르게 차단 가능
  - Closed, Open, Half-Open 상태 전이로 장애 차단과 회복 감지 지원

- RDB Fallback
  - 캐시 장애 상황에서도 조회 기능 유지 가능
  - Redis 회복 전까지 최소 기능을 유지하는 우회 경로
  - fallback 자체가 2차 장애를 만들 수 있어 bulkhead와 짧은 DB timeout 필요

## 검증

### 테스트 설정
- Redis 연결 실패와 timeout 상황 재현
- `slidingWindowSize`, `minimumNumberOfCalls` 조정
- Closed → Open → Half-Open 상태 전이 확인
- 100 TPS는 학습용 조회 서비스의 피크 모델로 50명 동시 사용자와 사용자당 2 rps 기준 산정
- 장애 구간 3분 동안 100 TPS 조회 부하 적용
- fallback 동시 실행 수 제한 전후 p99 응답 시간 비교

### 측정 결과
- Redis timeout 반복 구간 p99 1.2초 수준 확인
- Circuit Open 이후 p99 180ms 수준으로 감소
- fallback 동시 실행 수 제한으로 RDB 커넥션 고갈 방지

## 핵심 코드
- Cache Miss는 RDB 조회 후 Redis 적재
- Circuit Open은 Redis 생략 후 RDB 직접 조회

```java
@CircuitBreaker(name = REDIS_CIRCUIT_BREAKER, fallbackMethod = "fallbackOnRead")
public ArticleQueryModel read(Long articleId) {
    return articleQueryModelRepository.read(articleId).get();
}

private ArticleQueryModel fallbackOnRead(Long articleId, ArticleCacheMissException e) {
    return fetch(articleId).orElseThrow();
}

private ArticleQueryModel fallbackOnRead(Long articleId, CallNotPermittedException e) {
    log.warn("[REDIS CIRCUIT BREAKER] : OPEN");
    return readRDB(articleId);
}
```

## 배운 점
- fallback은 우회 경로이면서 새로운 병목 후보
- Circuit Open 효과는 상태 전이보다 장애 구간 latency 수치로 검증 필요
