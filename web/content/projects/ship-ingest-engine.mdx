## 개요
- Ethernet Converter 기반 통신 환경에서 선박 운항·안전 센서 데이터 수집
- Netty 기반 비동기 TCP 수집 구조로 메시지 프레임 디코딩 처리

## 문제

- 기존 비즈니스 트래픽(선원 위치, 각종 센서, 서비스 기능) 처리 중 컨버터 유입 트래픽 추가
- Ethernet Converter를 통해 선박 위치, 방향, 화재 센서 데이터 신규 유입
- 운영 장비 예산과 기존 인프라 제약으로 SSD 대신 HDD 기반 스토리지 사용
- HDD 기반 스토리지의 낮은 IOPS(초당 처리 가능한 입출력 횟수)로 인한 Write 처리량 제약
- 기존 비즈니스 트래픽과 단건 Insert 로그 적재가 겹치며 DB I/O Wait 18%까지 상승

## 해결 전략

- 수집된 로그를 ConcurrentLinkedQueue에 적재한 뒤 배치 작업에서 주기적으로 비우는 구조 적용
- 단건 Insert를 주기·크기 기반 Batch Insert로 전환
- 큐 적재량 hard limit 10,000건 설정 후 초과 시 TCP 채널 read 일시 중지로 backpressure 적용
- 큐 적재량 7,000건 경고 임계치에서 flush 주기 단축과 운영 알림 수행
- 프로세스 장애 시 미반영 메모리 로그 유실 가능성을 운영 리스크로 분리

## 기술 선택 이유

- Batch Insert
  - HDD 환경에서 단건 Insert 반복 시 랜덤 Write 비용 증가
  - 묶음 쓰기로 DB 왕복 횟수와 디스크 I/O 부담 감소

- ConcurrentLinkedQueue
  - 수집 스레드와 배치 저장 스레드 간 Lock 경합을 줄이기 위한 메모리 큐
  - 데이터 수신 흐름과 DB Write 흐름 사이에서 버퍼 역할 수행
  - 짧은 버스트 흡수 목적의 임시 버퍼로 사용하고 장기 적체 저장소로 사용하지 않는 전제
  - 센서 시스템의 Queue 1 정책은 최신 상태 전송 목적, 수집 엔진의 메모리 큐는 DB Write 흡수 목적
  - hard limit 도달 시 Netty `autoRead(false)`로 생산 속도를 낮추는 제어 지점 확보

## 튜닝 기준

- Batch Size
  - 100건, 300건, 500건 단위로 비교
  - 100건은 DB 왕복 횟수 감소 효과가 작고, 500건은 메모리 적재 시간 증가
  - 300건 기준에서 Write 횟수와 반영 지연 사이의 균형 확인

- Flush 주기
  - 1초 주기 Flush 설정
  - 피크 구간은 Batch Size 300건 도달 시 우선 반영
  - 저유입 구간은 1초 주기로 잔여 로그 반영

- Queue 상한
  - 평균 로그 크기 600B 기준 10,000건 적재 시 약 6MB 수준으로 hard limit 설정
  - 7,000건 초과 시 flush 주기 1초에서 300ms로 단축
  - 10,000건 도달 시 TCP read 일시 중지 후 5,000건 이하 회복 시 재개
  - 지속 적체 구간은 HDD Write 병목으로 판단하고 DB 저장 경로 분리 검토

## 검증

### 테스트 설정
- Locust 기반 부하 테스트 수행
- 기존 비즈니스 트래픽 1,000 TPS는 선원 300명 위치 갱신, 센서 100개, 서비스 기능 호출을 합산한 피크 모델
- 선원 위치, 각종 센서, 서비스 기능 요청 동시 처리 상황 재현
- Ethernet Converter 유입 트래픽으로 선박 위치, 방향, 화재 센서 데이터 추가
- 컨버터 수집 로그 300 TPS는 선박 위치 10Hz, 방향 10Hz, 화재·안전 센서 burst를 합산한 여유 피크
- 3분 ramp-up 후 10분간 지속 부하 수행
- 단건 Insert와 Batch Insert의 DB I/O Wait 비교
- 큐 적재량, flush 처리량, 미반영 로그 수를 함께 관찰
- hard limit 도달 시 read 중지와 재개 동작 확인

### 측정 결과
- DB I/O Wait 18%에서 3%로 감소
- HDD 환경에서도 수집 지연 없이 안정적 Write 처리
- 지속 부하 구간에서 큐 적재량이 flush 주기 내 해소되는지 확인
- 과부하 주입 구간에서 큐 10,000건 초과 전 TCP read 중지 확인

## 배운 점
- 메모리 큐는 버스트 흡수 장치일 뿐 내구성 보장 장치가 아님
- 최신 상태 전송 큐와 로그 적재 버퍼는 목적에 따라 상한 정책이 달라짐
- 알림은 관측 수단이고 OOM 방지는 hard limit과 backpressure가 담당
